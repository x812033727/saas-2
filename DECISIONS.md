# 架構決策記錄（ADR）

## 修法選 `Query(ge=1, le=N)` 回傳 422，三端點行為一致，不用 clamp 也不拆分上下界處理。
- 時間：2026-07-17 19:07
- 理由：FastAPI 框架層統一校驗，測試邊界只需測 422/200，不用鋪 500+ 列；單一機制讓 API 契約可預測。
- 否決方案：「下界 ge=1 拒絕、上界 min clamp」混合模式——同一參數兩種語意，維護難且測試代價不對稱。

## `limit` 簽章改為 `limit: int = Query(<default>, ge=1, le=<max>)`，各端點參數如下——runs 預設50/上限200、stats 預設20/上限100、alerts 預設100/上限500。
- 時間：2026-07-17 19:07
- 理由：與原 `min(limit,N)` 上限完全對齊，預設值不變，不破壞既有 API 契約。

## 移除 `.limit(min(limit, N))`，改為裸 `.limit(limit)`，並在同行加行內註解 `# upper bound enforced by Query(le=N)` 標明值域由框架層保證。
- 時間：2026-07-17 19:07
- 理由：移除冗餘防禦避免語意混亂；加註解防止半年後有人看到裸 `.limit(limit)` 誤以為漏了防禦又補回 `min`。
- 否決方案：保留 `min(limit, N)` 作雙重保險——框架已保證值域，留著反而是技術負債，且掩蓋了 422 路徑應涵蓋的測試。

## 唯一行為 breaking change——`limit` 超過上限（如 limit=9999）從靜默截斷改為 422，需在 commit message 明記。
- 時間：2026-07-17 19:07
- 理由：內部平台 API，無任何現存測試依賴超上限靜默截斷，風險可接受；明記讓日後追查有跡可循。

## 測試斷言用精確值 `len == 2`，不用 `len <= 2`。
- 時間：2026-07-17 19:07
- 理由：弱斷言（<= 2）在 limit 壞掉回 0 列時也會通過，無法真正驗收「limit 生效且上限正確」。鋪 3~5 列、傳 limit=2，斷言精確列數。

## 測試覆蓋三端點 × 三邊界（limit=-1、limit=0、limit=N+1）→ 斷言 422；各端點再加一個合法值（limit=2，鋪 3~5 列）→ 斷言 200 且 `len == 2`。
- 時間：2026-07-17 19:07
